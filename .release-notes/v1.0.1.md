# v1.0.1 — On-demand HVVH validation + path-to-regexp v8 wildcard fix

Two small fixes on top of v1.0.0 (GA foundation):

## What's new

### On-demand `validate-hvhh` endpoint pair (customer + vendor)

The CFO / AP team wanted "after a year of edits, is this customer's hvhh still valid?
Hit a button and find out." v1.0.0 had fail-soft A1-Validator drift detection on
writes (POST /customers, /vendors, /invoices, /vendor-bills, /crm/contacts,
/crm/leads, /pos/sales) — but no read-time check.

Now:
- `POST /api/finance/customers/:id/validate-hvhh`
- `POST /api/finance/vendors/:id/validate-hvhh`

Each loads the live hvhh from the DB (not a denormalized column — same anti-pattern
lesson as v0.9.0's POS drift detection) and asks A1-Validator to verify it now.

Response always `200`:
```json
{
  "customer_id": 42,
  "hvhh": "00123456",
  "ok": true,
  "normalized": "00123456",
  "_via": "local-regex"  // or "a1-validator"
}
```

Invalid hvhh returns `200` with `ok: false` (the answer is "no, this is broken").
Never throws — the caller is verifying, not writing. `404` only when the row
doesn't exist.

Same perm gate as the 360 endpoints (`finance.customer.read` / `finance.vendor.read`).
Tenant-scoped via `requireTenant`.

### Path-to-regexp v8 wildcard fix

v1.0.0 included Wave 46 (commit 387b3ce) which translated Fastify `:name(*)`
wildcards to unnamed `*` for "cross-Express-version compat". But path-to-regexp v8
(what our env actually ships) **explicitly rejects unnamed `*`** — the splat
requires a name. Result: every Wave 46 route that used `:name(*)` (e.g. the rbac
`field-policies` + `record-rules` PUTs) was broken at server boot.

Reverted to `*name` syntax (works in path-to-regexp v6+). Handler-side fallback
to `req.params[0]` (added in 387b3ce) is still in place — covers the older
path-to-regexp v0.x where the splat lands at `params[0]` instead of `params[name]`.
