# v0.6.0 — A1-Validator integration across customer + vendor + invoice

The first release that ties HHVH (Armenian tax ID) validation to the [A1-Validator](https://github.com/Armosphera/A1-Validator) HTTP service end-to-end. Three create endpoints now share the same fail-soft pattern.

## What's new

- **`POST /api/finance/customers`** — validates `hvhh` via `assertValidHvhhAsync`. New `server/finance/hvhh-validator.js` wraps `lib/a1-validator-client.js` with a local `^\d{8}$` regex fallback.
- **`POST /api/finance/vendors`** — same wrapper, same pattern. Vendor TIN is validated by `assertValidVendorHvhhAsync`.
- **`POST /api/finance/invoices`** — re-validates the referenced customer's `hvhh` at invoice-create time. Catches drift: a customer's HVVH could have become invalid since the customer was created. The FK check now reads `SELECT id, hvhh FROM finance.customers` in one query, and the wrapper rejects on invalid HVVH.

## Fail-soft contract (3-tier)

1. `A1_VALIDATOR_URL` unset → the client is disabled; the local regex (`^\d{8}$`) is the only enforcement. No HTTP calls, no latency.
2. `A1_VALIDATOR_URL` set but service unreachable → falls back to the same local regex. The endpoint stays up.
3. `A1_VALIDATOR_URL` set + service reachable → calls the service.
   - `ok: true` → proceed.
   - `ok: false` → 400 with the service's error message.

The local regex is always the last-line-of-defense. The A1-Validator service is an enhancement, never a requirement.

## Upgrade notes

No breaking changes. Existing customers/vendors/invoices continue to work — the A1-Validator wrapper is opt-in via `A1_VALIDATOR_URL`. If you don't set the env var, behavior is unchanged (local regex only).

To enable the full validation:

```bash
export A1_VALIDATOR_URL=http://a1-validator:8000  # or https://your-a1-validator.example.com
npm start
```

See the README's "A1-Validator integration" section for details.
