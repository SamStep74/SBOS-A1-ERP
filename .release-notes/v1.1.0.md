# v1.1.0 — AI advisory + mutation handoff, HR status, scheduled reports, scheduler, audit search

Six waves of work on top of v1.0.1. This release closes the
advisory → mutation handoff for AI agents (suggest → apply
merge), ships the full HR employee-status lifecycle, adds
scheduled report runs with a real cron-driven worker, and
adds full-text search to the audit log.

## What's new

### AI agents (W93-1, W94-1, W99-1) — closed loop
The data quality module is now full-cycle. The CFO can:

1. **W93-1**: `GET /api/finance/ai/data-quality` — see the
   per-module quality score (customers / vendors / employees
   / invoices).
2. **W93-1**: `GET /api/finance/ai/duplicates` — see the
   specific duplicate customer groups (by hvhh or by name).
3. **W93-1**: `GET /api/finance/ai/hvhh-drift` — see the
   invoices whose snapshot hvhh differs from the current
   customer.hvhh.
4. **W94-1**: `GET /api/finance/ai/merge-candidates` — for
   each duplicate group, propose a primary + secondary
   merge plan with invoice + payment re-assignment counts.
5. **W94-1**: `GET /api/finance/ai/alerts?threshold=80` —
   severity-sorted alerts (critical / warning / info) for
   issues that exceed the threshold.
6. **W99-1 (NEW)**: `POST /api/finance/ai/apply-merge` —
   actually apply the merge. Re-assigns the secondary's
   invoices + payments to the primary, archives the
   secondary, and writes an audit log row. The advisory
   says "do this"; the mutation does it.

### HR basics (W91-1, W95-1) — full lifecycle
The HR module now has the full employee-status lifecycle.
State machine:

```
active → on_leave, suspended, terminated
on_leave → active, suspended, terminated
suspended → active, terminated
terminated → (terminal)
```

- `POST /api/finance/hr/employees/:id/suspend` (state-machine guard active → suspended)
- `POST /api/finance/hr/employees/:id/reactivate` (suspended → active)
- `POST /api/finance/hr/employees/:id/on-leave` (active → on_leave)
- `POST /api/finance/hr/employees/:id/terminate` (on_leave → terminated)

Audit fields (`suspended_at`, `on_leave_at`, `on_leave_until`,
`termination_reason`) are stamped at transition time and NOT
cleared on subsequent transitions (preserves history).

### Phase 3 reporting (W92-1, W96-1, W97-1) — scheduled runs
The reporting module now has the full scheduled-reports
pipeline.

- **W92-1**: drill-down routes (ar-aging-bucket,
  revenue-trend, customer-breakdown).
- **W96-1**: data + API for scheduled reports (2 new tables,
  6 pure functions, 5 HTTP routes).
- **W97-1 (NEW)**: the scheduler worker. `computeNextRunAt`
  with proper Vixie dom/dow semantics. `dispatchReport`
  injects the right report function. `tickOnce` runs the
  worker once. `startScheduler` wires it to setInterval
  (default 60s tick). Boot wires it via `opts.scheduler`
  (can disable via `opts.scheduler = false`).

### Audit log (Wave 50) — full-text search
`GET /api/finance/audit?q=search-term` — full-text search
across the audit log (resource type, action, entity id,
user, IP, user agent). Replaces the need for SQL queries
when investigating "what happened to customer X last week?".

### v1.0.1 carryover
The v1.0.1 on-demand `validate-hvhh` endpoints are still
shipping in v1.1.0 (we never released v1.0.1 as a final
release — the 1.0.1 tag exists, but v1.1.0 supersedes it
with more features).

## Operational notes

- **Scheduler autostart**: by default, the server starts the
  scheduler worker. Disable via `opts.scheduler = false` in
  `createApp` (useful for tests + read-only CLI tools).
- **AI apply-merge audit trail**: every merge writes a row
  to `finance.customer_merge_log` (append-only). The
  `archived` column on `finance.customers` is the
  soft-delete flag (set to 1 on the merged-away secondary).
  Both fields are queryable for forensics.
- **HR state machine guard**: every status transition
  validates the source state. `terminated` is terminal
  (no transitions out). Re-hires require a new employee
  record.

## Migration notes for existing deploys

- 3 new migrations apply on top of v1.0.1's 25.
- No breaking schema changes — all new columns / tables
  are additive.
- The scheduler autostart can be disabled per-deploy if
  you don't want report runs firing automatically.
