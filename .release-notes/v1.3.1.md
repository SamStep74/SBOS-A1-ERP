# v1.3.1 — session activity log (Wave 55)

One commit on top of v1.3.0. The sessions table holds current
state; this release adds the lifecycle event log so the
operator can see who logged in when, who got revoked, and
who did it from where.

## What's new

### Session activity log (Wave 55)
A new `sbos_session_events` table tracks every session
lifecycle event. The existing `audit_events` table is for
security/governance (logins as `resource=user:X`,
`action=login`); the new table is for session lifecycle
specifically (login, logout, revoked). Different concerns,
different shapes — kept separate.

**Schema (migration `0003_session_events.sql`):**
- `sbos_session_events(id, session_id, user_id, tenant_id,
  event_type, ip, user_agent, payload_json, created_at)`
- Index on `(session_id, created_at DESC)` for per-session
  queries
- Index on `(user_id, created_at DESC)` for "my activity"
- Index on `(created_at DESC)` for global time-range

**Pure functions in `server/auth-sessions.js`:**
- `recordSessionEvent(db, { sessionId, userId, tenantId,
  eventType, ip, userAgent, payload })`
- `listSessionEvents(db, sessionId, { limit })`
- `listUserSessionEvents(db, userId, { limit })`

**Endpoints:**
- `GET /api/auth/sessions/events` — self-service "my activity"
- `GET /api/rbac/sessions/:id/events` — admin, tenant-checked

**Hooks (events recorded automatically):**
- `login()` → `event_type=login` on success
- `logout` → `event_type=logout` on revoke
- `admin DELETE /api/rbac/sessions/:id` → `event_type=revoked`
  with `payload.source=admin` + `revoked_by`
- `self /api/auth/sessions/:id/revoke` → `event_type=revoked`
  with `payload.source=self`
- `seedSessionForAdmin` (boot) → `event_type=login` with
  `payload.method=boot-seed`

### Live-swap bugfix (4th in the family)
Four more `db`-captured closures in `server/index.js`
(login, logout, listMySessions, revokeMySession,
revokeAllMySessions) updated to read `dbRef.current`.
The captured-const bug family across waves 52/53/54/55
is now closed.

## Operational notes

- **My-activity** is a self-service endpoint. The user
  can see "I logged in from IP X at time Y" without an
  admin intermediary.
- **Admin per-session events** is tenant-checked — an
  admin from tenant 0 cannot see session events from
  tenant 7's session, even with the session id.
- **Boot-seed login event** lets the operator see the
  initial admin session in the event log (without it,
  you'd have an admin "logged in forever" with no
  originating event).
