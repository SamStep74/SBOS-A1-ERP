# v1.3.0 — undo customer merge, live backup restore, account lockout ops, scheduler concurrency, inventory adjustment reasons

Eleven commits on top of v1.2.0. This release closes the
advisory → mutation → undo round-trip for AI agents
(suggest → apply → undo), ships the destructive half of
backup (restore from snapshot), adds the operational
counterparts to account lockout (observability + force-lock),
makes the scheduler safe under slow ticks, and makes
stock adjustments auditable with mandatory reasons.

## What's new

### AI agents wave 4 (W102-1) — undo customer merge
The W99-1 `applyCustomerMerge` was destructive. The new
`undoCustomerMerge` is its inverse: re-assign invoices +
payments back to the secondary, un-archive it, and
record an "undone" entry in the merge log.

- `POST /api/finance/ai/undo-merge` (body: merge_id, reason)
- The merge log row is the source of truth for both the
  original merge AND the undo. The migration
  `0030_customer_merge_undo.sql` adds 5 columns to
  `customer_merge_log` (reassigned_invoice_ids,
  reassigned_payment_ids, undone_at, undone_by_user_id,
  undone_reason) so the undo can know which records to
  restore.

### Live restore-from-backup (Wave 52) + live-swap bugfix
The W47 backup endpoint only saved backups. W52 adds
`POST /api/rbac/backup/restore` — the destructive
complement that replaces the live sqlite file in place
after validating the chosen snapshot.

The restore requires a refactor of the RBAC routes to
read the live db handle via a mutable `dbRef` (not a
captured const). The auth middleware got the same
refactor in W53. The pgAdapter got it in W54. The
pattern is now uniform: any long-lived resource (db
handle) lives in `dbRef = { current: ... }` and every
consumer reads `dbRef.current` per call.

### Account lockout observability + force-lock (Wave 53)
The W38 auto-lockout was a black box. W53 adds:
- `GET /api/rbac/users` with `?locked=true` filter —
  list users with pagination
- `POST /api/rbac/users/:userId/lock` — force-lock a
  user with a mandatory free-text reason (recorded in
  the audit log)

The 1-hour default lock is short enough that the
existing unlock route can clear it if the operator
walks away, but long enough to block access in the
meantime.

### Phase 3 reporting wave 6 (W103-1) — run-now admin
The scheduler runs reports on a cron. What if the CFO
wants to run a report NOW (without waiting for the next
cron tick)? New:
- `POST /api/finance/reports/schedules/:id/run-now` —
  dispatches the report immediately + records the
  execution with `triggered_by = 'manual'`

### Phase 3 reporting wave 7 (W104-1) — scheduler concurrency
The scheduler is a `setInterval` that runs every 60s. If
a tick takes longer than 60s (e.g. slow SMTP send), the
next tick fires while the previous is still running.
Without a guard, this causes duplicate execution rows +
duplicate emails.

The fix: an `inProgress` flag set when a tick starts and
cleared in `.finally` (always runs). Concurrent ticks are
skipped. Plus 8 live-getter metrics (totalTicks,
skippedTicks, completedTicks, erroredTicks, inProgress,
lastTickAt, lastTickDurationMs, lastTickError) exposed
via `handle.metrics` for observability.

### Inventory adjustment reasons (Wave 54)
Every `POST /api/finance/stock/adjust` now requires a
`reason` (5-500 chars) and a `reason_category` (one of
damage, loss, found, correction, recount, writeoff,
other). New `GET /api/finance/stock/adjustments` endpoint
with `?category`, `?itemId`, `?locationId`, `?since`,
`?limit` filters for the "why did this number change?"
audit question.

### Catalog pricing rules (f7fba19 cherry-pick)
The catalog v2 wave 3d wires the pricing-rule pure
functions (W80-1) into 3 HTTP endpoints. Plus 2 new
perm keys (`finance.pricing_rule.read` + `...create`) on
the FinanceOperator set.

## Operational notes

- **AI merge undo** is bounded by the merge log. Pre-W102-1
  merges don't have the ID lists stored, so they can't be
  undone — only post-W102-1 merges can.
- **Live restore** does a `VACUUM INTO` of the current
  live db before swap, so the operator has a pre-restore
  snapshot to roll back to if the chosen backup turns
  out to be wrong.
- **Scheduler concurrency guard** is automatic — no config
  needed. If you want to observe the scheduler's health,
  read `handle.metrics` from the boot handle.
- **Account force-lock** writes to the audit log so the
  operator can see who locked whom and why.
- **Stock adjustment reason** is mandatory — old API calls
  that omit the field get a 400. Existing test seeds need
  to be updated to include the reason.
