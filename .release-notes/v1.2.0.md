# v1.2.0 — apply customer merge, audit search, backup listing, SMTP + webhook notifications

Five waves of work on top of v1.1.0. This release closes
the AI advisory → mutation handoff for customer merges
(suggest → apply), adds full-text search to the audit log,
exposes backup file listing + validation, and ships both
SMTP AND webhook notification transports for the scheduler.

## What's new

### AI agents wave 3 (W99-1) — apply customer merge
The W94-1 advisory said "merge these two customers". The
new `POST /api/finance/ai/apply-merge` actually does it:

- Re-assigns the secondary's invoices + payments to the
  primary
- Archives the secondary (soft-delete via `archived` flag)
- Writes an append-only audit row to
  `finance.customer_merge_log` (primary_id, secondary_id,
  invoice/payment counts, applied_by, reason, timestamp)

The advisory → mutation handoff closes the W93-1 + W94-1
loop. From a CFO's perspective: "show me the data quality
issues" → "show me the merge plan" → "apply the merge" → audit
log records everything.

### Audit full-text search (Wave 50)
`GET /api/finance/audit?q=search-term` — full-text search
across the audit log. Replaces the need for SQL queries
when investigating "what happened to customer X last week?".

### Backup listing + validation (Wave 51)
The Wave 47 database backup endpoint stored backup files
on disk but didn't expose them. Wave 51 adds:
- `GET /api/finance/backups` — list backup files with
  size, timestamp, status
- `GET /api/finance/backups/:filename/validate` — verify
  the backup is a valid sqlite database

### SMTP notifications (W101-1) — sibling session
The `sendNotificationEmail` stub is replaced with a
hand-rolled SMTP transport. The scheduler can now actually
deliver emails (no more console.log only). For
self-contained deployments with no SMTP server, the
"capture" mode writes emails to disk instead.

### Webhook notifications (W98-1) — this session
`POST` the report result as JSON to a configured URL. Works
with Slack / Discord / Microsoft Teams bots, Sendgrid's
HTTP API, or any service that accepts JSON POST. HMAC-SHA256
signing via the `X-SBOS-Signature` header for authentication.

New: `notify_webhook_url` + `notify_webhook_secret` columns
on `finance.report_schedules`.

## Operational notes

- **AI apply-merge audit trail**: every merge writes a row
  to `finance.customer_merge_log` (append-only). The
  `archived` column on `finance.customers` is the
  soft-delete flag (set to 1 on the merged-away secondary).
  Both fields are queryable for forensics.
- **SMTP transport**: opt-in. Set `SBOS_SMTP_HOST`,
  `SBOS_SMTP_PORT`, `SBOS_SMTP_USER`, `SBOS_SMTP_PASS`
  env vars to enable. Otherwise the scheduler falls back
  to "capture" mode (writes emails to `sbos-emails/`).
- **Webhook transport**: opt-in per-schedule. Set
  `notify_webhook_url` on the schedule. The signature
  secret is per-schedule too (different schedules can
  have different webhook consumers).
- **Both transports fire on the same tick**. If a schedule
  has both `notify_email` AND `notify_webhook_url`, the
  scheduler fires both. Either failure is non-fatal.
