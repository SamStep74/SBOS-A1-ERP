# v1.4.0 — invoice attachments (Wave 56) + login rate limiting (Wave 57)

Two new waves on top of v1.3.1. Both close gaps that surfaced
under real use: operators needed to attach supporting documents
to invoices (signed PDFs, photos of received goods, customs
declarations) and the auth path needed to slow down brute-force
attempts on the boot-minted admin token.

## What's new

### Wave 56 — invoice document attachments
Per-invoice document attachments. Files live on disk under
`$SBOS_ATTACHMENTS_DIR` (default `./attachments`); the DB stores
metadata only (size, sha256, mime type, storage path, uploader).

**Schema (migration `0032_invoice_attachments.sql`):**
- `invoice_attachments(id, tenant_id, invoice_id, filename,
  mime_type, size_bytes, sha256, description, storage_path,
  uploaded_by, uploaded_at, deleted_at)`
- Index on `(tenant_id, invoice_id, uploaded_at DESC)` for
  per-invoice list
- Index on `(tenant_id, deleted_at)` for the soft-delete filter

**Pure functions in `server/finance/attachments.js`:**
- `addAttachment(pgAdapter, { tenantId, invoiceId, buffer,
  filename, mimeType, description, uploadedBy, attachmentsDir })`
- `listAttachments(pgAdapter, tenantId, invoiceId, { limit })`
- `getAttachment(pgAdapter, tenantId, attachmentId)`
- `deleteAttachment(pgAdapter, tenantId, attachmentId)` (soft)
- `readAttachmentBytes(attachment)` — verifies size drift
  between DB and disk
- `normalizeAttachmentFilename`, `isValidMimeType`,
  `computeStoragePath`, `resolveAttachmentPath`, `sha256Hex`

**Endpoints:**
- `POST   /api/finance/invoices/:id/attachments` — raw body +
  `x-filename` / `x-mime-type` / `x-description` headers
- `GET    /api/finance/invoices/:id/attachments` — list metadata
- `GET    /api/finance/invoices/:id/attachments/:attId` — raw
  bytes with `Content-Disposition: attachment; filename=...`
- `DELETE /api/finance/invoices/:id/attachments/:attId` — soft-delete

**Perm keys:** `finance.invoice.attach` (write) and
`finance.invoice.attach.read` (read), both bound to
`FinanceOperator`.

**Safety:**
- Filename blocklist rejects `.exe`, `.dll`, `.bat`, `.cmd`,
  `.sh`, `.msi`, `.com`, `.scr`, `.vbs`, `.ps1`, `.cpl`, `.jar`
- Mime whitelist (~80 known types) via `isValidMimeType`
- Sha256-hashed storage path — original (unsafe) filename
  preserved in DB for display
- Raw body read off the request stream (bypasses the 1mb JSON
  body cap)

### Wave 57 — login rate limiting
Brute-force defense on `POST /api/auth/login`. After 5 failed
attempts in a 15-minute window, the account is locked for
15 minutes (returns 423 Locked). Successful login resets the
counter. Lockout state persists in `users.locked_until` so a
server restart does not bypass the throttle.

**Schema:** no new migration — reuses `users.failed_logins`
and `users.locked_until` from earlier waves.

**Functions in `server/auth-login.js`:**
- `recordFailedLogin(db, username)` — increments counter,
  sets `locked_until` if threshold crossed
- `resetFailedLogin(db, username)` — on successful login
- `isLocked(db, username)` — predicate for the 423 response
- `unlockAccount(db, username)` — admin force-unlock (reuses
  the Wave 49 admin endpoint)

**Endpoint:** `POST /api/auth/login` returns 423 with
`{error: "locked", message: "..."}` on lockout, 401 on bad
creds, 200 + token on success.
