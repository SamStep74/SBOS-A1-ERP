# v1.0.0 — GA foundation

The first stable release of SBOS-A1-ERP. This release marks the end of the Phase-1 ERP foundation (RBAC + finance + audit + POS + lots/serials + recall) and the start of Phase 2 (drift-detected customer/vendor/CRM/AR/AP data integrity via the A1-Validator).

## What's in v1.0.0

### Five SBOS endpoints now use the A1-Validator HTTP service

The Armenian HHVH (8-digit tax ID) is the primary identifier across SBOS — used by customers, vendors, contacts, leads, and POS sales. v1.0.0 wires the A1-Validator service into every create / write path that takes or references a customer's HVVH:

| Endpoint | Since | Pattern |
|---|---|---|
| `POST /api/finance/customers` | v0.6.0 | Direct field validation (hvhh is optional) |
| `POST /api/finance/vendors` | v0.6.0 | Direct field validation (hvhh is optional) |
| `POST /api/finance/invoices` | v0.6.0 | **Drift detection** — re-fetches customer.hvhh at invoice-create time |
| `POST /api/finance/vendor-bills` | v0.7.0 | **Drift detection** — re-fetches vendor.hvhh at bill-create time |
| `POST /api/finance/crm/contacts` | v0.7.0 | Direct field validation (hvhh is optional, for self-employed contacts) |
| `POST /api/finance/crm/leads` | v0.8.0 | Direct field validation (hvhh is optional, for prospective companies) |
| `POST /api/finance/pos/sales` | v0.9.0 | **Drift detection** — re-fetches customer.hvhh at sale-create time (walk-in sales skip) |

The drift-detection pattern is the real value: when invoice/vendor-bill/POS-sale is created, the live customer's HVVH is re-fetched from `finance.customers` (not from a denormalized column on the dependent table), so a customer-edit that invalidated the HVVH between the original create and the dependent write is caught at write time.

The A1-Validator service runs as a separate HTTP process (configurable via `A1_VALIDATOR_URL`). When the service is unreachable, the wrapper falls back to a local regex (`^\d{8}$`). The endpoint stays up; the A1-Validator is an enhancement, never a requirement. See [docs/CI.md](https://github.com/Armosphera/SBOS-A1-ERP/blob/main/docs/CI.md) and the README's "A1-Validator integration" section for the contract.

### Phase-1 ERP feature set

The v1.0.0 GA foundation includes:
- **RBAC** — 28 roles, 445 permissions, 74 perm sets; tenant-scoped (Wave 28); perm-gated per route (Wave 27); audit-logged

- **POS** — registers, shifts, sales (open → completed → voided/refunded), sale lines, payments
- **Inventory** — lots + serials, stock moves, weighted-average cost
- **Recall** — lot + serial cascade (regulatory compliance for batch-tracked goods)
- **Drift detection** — 5 endpoints validate TIN drift via A1-Validator
- **Local CI** — `npm run check` + `scripts/deploy-smoke.sh` (repo is private; GH Actions disabled in commit acc3c0f)

## Upgrade notes

**No breaking changes.** All previous endpoints behave the same when `A1_VALIDATOR_URL` is unset (local regex fallback). When the A1-Validator service is reachable, the 5 drift-detection endpoints re-validate the customer's HVVH at write time.

**Migration behavior on existing deploys:**
- Migrations 0017-0023 were added between v0.8.0 and v1.0.0. Existing deploys will see these as new and apply them (safe with `IF NOT EXISTS` / `CREATE TABLE IF NOT EXISTS`).
- The two migration renames that shipped in v0.8.0 + v0.9.0 (`0013_pos_basics → 0017_pos_basics`, `0009_replenishment → 0019_replenishment`, etc.) preserve the migration_history table entries for any deploy that ran the original filenames — those rows are kept and the renamed files are seen as new (safe with `IF NOT EXISTS`).
