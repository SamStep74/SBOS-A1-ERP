## What's shipped (Phase 0 + Phase 1 ERP)

### Phase 0 — foundation (waves 1-15, ~5 days of work)

- **Catalog-driven RBAC** —
  315 permission keys, 27
  system roles, 39 permission
  sets, runtime guards (perm +
  FLS + RLS + session +
  impersonation). The
  foundation of every
  endpoint.
- **Bootable HTTP server**
  (`bin/sbos-server.mjs`) —
  mints an admin session
  token on first boot,
  seeds the RBAC catalog,
  links the admin user to the
  Admin role.
- **Multi-tenant** — every
  endpoint is tenant-scoped
  (the `requireTenant` guard
  is a hard gate).
- **Deploy story** — 3 paths
  (bare metal / Docker /
  systemd+pm2) + backup +
  smoke + token helper. See
  [`docs/DEPLOY.md`](./DEPLOY.md).
- **CI** — `.github/workflows/ci.yml`
  runs lint, typecheck, test,
  boundary-check, deploy-smoke
  on every push.

### Phase 1 ERP (waves 16-17, ~1.5 days of work)

- **Inventory** (wave 16.1) —
  warehouses, stock
  locations, catalog items
  (products with SKU + unit
  cost + UOM), stock moves
  (receive / transfer /
  adjust), weighted-avg cost
  calculation. Migration
  `0007_inventory.sql`.
- **Purchase** (wave 16.2) —
  vendors, purchase orders,
  3-way match (PO → receive →
  vendor bill). Migration
  `0008_purchase.sql`.
- **Route wiring** (wave
  16.3) — 24 new endpoints
  with permission gates +
  audit logging. Deploy-smoke
  extended from 11 → 35
  endpoints.
- **README** (wave 16.4) —
  the Phase 1 ERP API surface
  section.
- **PO + delivery-note**
  (wave 17) — Armenian-first
  template engine (text +
  HTML rendering) + print
  routes for PO + delivery
  note.

### Open-core story (waves 14-15)

- **Dockerfile** —
  multi-stage build,
  `python:3.12-slim` base,
  tini as PID 1, non-root
  user. ~28MB final image.
- **systemd unit** —
  hardened (NoNewPrivileges,
  ProtectSystem=strict,
  ProtectHome, etc.).
- **pm2 ecosystem** — alt
  process manager for non-
  systemd hosts (macOS dev
  boxes, some cloud VMs).
- **Token persistence** — the
  admin session token is
  written to a file (mode
  0600) for multi-host
  deploy; new hosts that
  mount the same data dir
  reuse the same token.

- **Desk** — ticketing /

## What's blocked

- **Multi-host / K8s deploy**
  — the current deploy is
  single-node only. The
  admin token file pattern
  works for multi-host (a
  shared volume gives all
  hosts the same token), but
  no operator runbook for
  K8s / ECS / Nomad exists
  yet. Customer-driven; not
  blocking v0.1.0.
- **Backup verification** —
  the cron-driven backup
  works (online-safe via
  `sqlite3 .backup`), but
  the restore-from-backup
  path is unverified. A
  follow-up plan should add
  a "restore drill" test
  (boot from a known-good
  backup; smoke-check).
- **Production pg adapter** —
  the `pg` adapter is wired
  (with the `stripFinancePrefix`
  helper + the `$/N → ?`
  translation), but the
  current production deploys
  use sqlite. A follow-up
  plan should verify the pg
  path against a real pg
  instance (a CI job that
  spins up pg + runs the
  smoke check).
